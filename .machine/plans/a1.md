# hub — Implementation Plan (Revision 2)

## Overview

`hub` is a Rust binary that replaces `mesh/mesh.mjs` as the machine's MCP coordination daemon. It serves the 8 verb contract defined in SPEC-COMM-001 (`/.machine/specs/comm-daemon/SPEC.md`) over stdio JSON-RPC, preserves the `.mesh/state.json` schema so in-flight runtime state survives the upgrade without migration, and adds a dynamic tool registry with MCP `tools/list_changed` notifications. The `plugin.json` entry keyed `mesh` is replaced by an entry keyed `hub` in the same commit. `mesh/mesh.mjs` and `mesh/test.mjs` are deleted in that same commit. All call sites that reference the old MCP prefix `mcp__mesh__` or `mcp__plugin_machine_mesh__` are updated in the same commit — no stale references are left on main.

The `.mesh/` state directory name is kept unchanged, intentionally, for data continuity. Every git worktree of a given repo shares one `.mesh/` store and the directory name is not a user-facing identity — only the daemon name and MCP tool prefix change.

---

## 1. Crate Layout

Create `hub/` at the repository root alongside `mesh/`, `board/`, etc. The crate is standalone — no workspace `Cargo.toml` is created in this change; that is a separate concern.

Module structure inside `hub/src/`:
- `main.rs` — entry point; resolves data directory, builds server, drives stdio loop
- `state.rs` — JSON schema, locking, loading, saving, atomic rename
- `mesh.rs` — all 8 verb methods on a `Mesh` struct with injectable clock
- `ulid.rs` — monotonic ULID generation in Crockford base32 matching Node output character-for-character
- `registry.rs` — DashMap-backed dynamic tool table and broadcast channel for change notifications
- `server.rs` — wires registry and mesh into the rmcp stdio server
- `error.rs` — unified error type

Dependencies: `tokio` (async runtime, full feature set), `rmcp` (MCP stdio server with notification support), `serde` and `serde_json` (state serialization and JSON-RPC parsing), `dashmap` (concurrent registry map), `ulid` crate (ULID generation in Crockford base32 format the spec requires).

## 2. State Schema

The `.mesh/state.json` schema is frozen. The Rust `State` type must produce and consume JSON that is structurally identical to what `mesh.mjs` writes. The implementation agent must not rename fields, add fields, remove fields, or change the format of any value.

The top-level object has seven keys: `roster`, `claims`, `messages`, `log`, `cursors`, `events`, and `fence_floor`. All seven must be present on every write even when their values are empty objects or empty arrays. This matches the `#emptyState()` fallback in the Node daemon.

Specific format rules that must be preserved exactly:
- Integer timestamps are Unix seconds stored as JSON numbers (not strings).
- ISO timestamps in event records, log entries, and verb return values are RFC3339 UTC at second precision with no millisecond fraction (e.g. `2026-06-14T12:00:00Z`). The Node daemon produces this with `.toISOString().replace(/\.\d{3}Z$/, "Z")`.
- Fields that may be absent in some records (such as `role` on a roster entry, `subject` and `reply_to` on a message, `note` on a holder) are stored as `null` in JSON, not omitted. Deserialization must handle both `null` and absent (use `serde(default)` on load), but serialization must always emit the key with `null` when the value is absent.
- The `fence_floor` object accumulates per-resource fence values when a claim record is fully retired and must survive round-trip through the Rust layer without loss.

Locking: the `mkdir`-as-mutex pattern from the Node daemon is replicated exactly. `std::fs::create_dir` on the lock path succeeds for exactly one process; spin with a 2 ms pause; steal the lock after 5 000 ms by removing it and retrying once. Write via temp file plus rename (atomic on POSIX same-filesystem). The lock is held only for the duration of a single transaction: load, mutate, save, unlock.

## 3. Mesh Verbs

The 8 verb contract is specified in SPEC-COMM-001 §4 (`/.machine/specs/comm-daemon/SPEC.md`). The implementation agent must read that document as the canonical verb-by-verb contract and implement to it. This plan does not restate the request/response shapes or field-level semantics — they are the spec's responsibility.

Behavioral invariants the test suite must cover, derived from the Node daemon:

`register` is idempotent for a live agent (same epoch, updated heartbeat) and bumps epoch when called after the agent has entered the `dead` liveness state. `roster` runs a sweep of expired claims before returning. Liveness transitions: alive while `now <= expires_at`; stale while `now <= expires_at + 30`; dead after that. Dead agents are hidden from roster by default and visible only when `include_stale` is true.

`claim` is idempotent for the current holder (renews lease, does not bump fence). Shared grants co-hold. Exclusive blocks shared and vice versa. The fence is monotonically increasing per resource. Dead agents' hold entries and queue tickets are reaped on the next claim or sweep touch. Queue promotion fires when holders drops to zero after a release or reap. `release` distinguishes between unknown claim IDs, claim IDs belonging to a different agent, and valid releases; it also handles cancelling a queued ticket without promoting.

`post` generates a ULID as the message ID and computes fanout as the roster size for broadcast, 0 for topic addresses, and 1 for direct. `inbox` does not advance the cursor. `read` advances the cursor only forward — never backward. Topic routing matches recipients of the form `topic:<name>` against the `topics` list in the request.

A `gc` CLI subcommand (not an MCP verb, matching the Node daemon's `mesh gc` subcommand) sweeps dead claims and drops TTL-expired messages from both `state.messages` and `state.log`.

## 4. Tool Registry

The registry is a live, mutable table of MCP tool descriptors maintained in memory and broadcast to connected clients when it changes. It exists alongside the 8 static mesh verbs; the mesh verbs are pre-registered at startup and never change.

The registry holds tool entries by name. Each entry carries a name, a description, and a JSON Schema object for the input. The registry exposes operations to register a new entry, unregister by name, and list all current entries sorted by name. A `tokio::sync::broadcast` channel carries zero-payload change tokens; any register or unregister operation sends a token.

A background task holds a receiver on that channel. When it receives a token it drains any additional pending tokens before sending a single `notifications/tools/list_changed` JSON-RPC notification on the server's outbound channel. This coalescing prevents a burst of registry changes from generating a proportional burst of notifications to the client.

The `tools/list` response is always sourced from the live registry, which always includes the 8 mesh verbs. The `capabilities` field in the `initialize` response must declare `tools: { listChanged: true }`. The server name in the `initialize` response is `hub`, version `0.4.0`.

## 5. MCP Server Wiring

The rmcp crate drives the stdio loop: it reads newline-delimited JSON-RPC from stdin, dispatches to handler functions, and writes responses to stdout. The implementation uses `server.serve_stdio().await` in the async main.

Method routing: `initialize` returns the handshake with `listChanged: true`; `notifications/initialized` and `ping` are acknowledged; `tools/list` returns the registry listing; `tools/call` dispatches by name to the matching mesh verb method or registry handler; unknown methods return a JSON-RPC error with code `-32000`. Tool call results are wrapped as `{ content: [{ type: "text", text: "<json>" }], structuredContent: <value> }` matching the Node daemon's wire format.

The notification pump task runs as a `tokio::spawn` concurrent with the stdio loop. EOF on stdin triggers graceful shutdown: the rmcp loop exits and the binary returns exit code 0, matching the Node daemon's `process.stdin.on("end", () => process.exit(0))`.

## 7. Build and Distribution

Hub adds a one-time build step to this repo. The `project.md` "Build / test / quality gate" section must be updated to include: `cargo build --release --manifest-path hub/Cargo.toml` as a required build step, and the hub binary existence check described in Section 10.

For local development: install the binary into the Cargo bin directory with `cargo install --path hub/`. The binary lands at `~/.cargo/bin/hub` (or `$CARGO_HOME/bin/hub`). This is suitable for rapid iteration.

For the plugin system: the `plugin.json` entry references `${CLAUDE_PLUGIN_ROOT}/hub/target/release/hub` with `args: ["mcp"]`. This path is the authoritative binary the plugin runtime uses; it requires a `cargo build --release` to have been run after clone or update.

The key in `mcpServers` changes from `mesh` to `hub`. The `version` field in `plugin.json` increments to `0.7.0`. The `hub/target/` directory is added to the repo root `.gitignore`. The compiled binary is not committed to git.

CI: no workflow exists today. When CI is added, the build step is `cargo build --release --manifest-path hub/Cargo.toml` and the artifact at `hub/target/release/hub` is what the plugin system needs. This is out of scope for the current change but documented in `project.md` for future reference.

## 12. Test Approach

Unit tests live in `hub/src/mesh.rs` under `#[cfg(test)]`. The clock is injectable via a function pointer on the `Mesh` struct. Each test creates a temporary directory, instantiates `Mesh` with a controlled clock, runs behavioral assertions, and drops the directory.

Tests must cover at minimum: register-plus-roster liveness state machine (alive, stale, dead, epoch bump on re-register after death); exclusive grant, deny, queue, promote, and fence monotonicity; idempotent renewal keeping claim ID and fence; shared co-holders with exclusive blocked; lease expiry freeing the lock; dead-agent claim self-heal; post plus inbox plus read cursor advance; inbox topic routing and broadcast; GC reclaiming expired messages.

State compatibility test: reads a `.mesh/state.json` fixture captured from a real Node daemon run, passes it through the Rust load function, round-trips through the Rust save function, and compares the output structurally as parsed JSON values (not byte strings) against the fixture.

Integration test: spawns the built binary as a child process, writes newline-delimited JSON-RPC to stdin, reads responses from stdout. Covers: `initialize` handshake checking `capabilities.tools.listChanged == true`; `tools/list` verifying all 8 verbs appear; register-roster round trip; claim-release round trip verifying fence increments; post-inbox-read round trip verifying cursor advance.

Pre-retirement gate: before the retirement commit, both test suites must be green: `node mesh/test.mjs` and `cargo test --manifest-path hub/Cargo.toml`.
