## Overview

This plan evolves `hub` from a per-session stdio MCP server into a singleton daemon
that runs one process per cwd, listens on port 7777, serves MCP over HTTP+SSE,
absorbs all 11 board verbs from `board/board.mjs`, and provides a live HTML board UI
over WebSocket. The `board` MCP entry and `board/board.mjs` are retired in the same
commit. `hub serve` becomes the primary entry point. The `hub mcp` stdio subcommand
is retained as a fallback.

The existing `.mesh/state.json` schema and the `.board/state.json` schema are both
frozen. The daemon name, port, and all call sites are updated together.

---

## 1. Transport Change

Hub currently ships as a stdio MCP daemon launched per-session via `hub mcp`. The
singleton model replaces this with `hub serve`, which binds port 7777 and keeps
running across sessions. `hub mcp` is retained as a fallback stdio subcommand for
local testing and for sessions that cannot reach the HTTP endpoint, but it is no
longer the primary transport wired into `plugin.json`.

The entry point in `main.rs` gains a second subcommand branch. `hub serve` calls an
async `serve(data_dir, port)` function that builds the axum router, binds
`0.0.0.0:7777`, and runs to completion. `hub mcp` continues to call `serve_stdio()`
exactly as today.

The `serve` function owns both the MCP session table and the board state, sharing
them across all connected clients via `Arc<RwLock<...>>` (reads dominate on board
queries; write lock only on mutations). The mesh state and board state are separate
locks; never held simultaneously to avoid cross-lock deadlock.

Startup behavior: on `hub serve`, the binary checks whether port 7777 is already
bound before attempting to bind. If a prior instance holds the port it exits cleanly
with a message indicating the daemon is already running.

## 2. MCP over HTTP+SSE

The MCP HTTP transport follows the MCP specification for Streamable HTTP.

`POST /mcp` receives a JSON-RPC request body. The server parses it, dispatches to
the appropriate handler, and returns the response as `application/json`. This is the
client-to-server path for all calls.

`GET /mcp` is the server-to-client SSE stream. Each connecting client is assigned a
session id. The client sends this id in subsequent `POST /mcp` requests via the
`Mcp-Session-Id` header. The server maintains a map from session id to a
`tokio::sync::mpsc` sender; responses and notifications for that session are written
to the sender and streamed as SSE events.

`tools/list_changed` notifications use a `tokio::sync::broadcast` channel. A
background task per session subscribes and forwards notifications onto that session's
outbound SSE stream. Coalescing (drain pending tokens before emitting one notification)
is carried forward from a1 unchanged.

The `initialize` response declares `capabilities: { tools: { listChanged: true } }`,
`serverInfo: { name: "hub", version: "0.8.0" }`, and returns the `Mcp-Session-Id`
in the response header.

Axum routes for the MCP transport:
- `POST /mcp` â€” parse body, dispatch, return JSON or push onto session SSE stream
- `GET /mcp` â€” SSE stream for this session

Both routes share the same `HubState` extractor carrying `Arc` handles to mesh
state, board state, session map, and broadcast channel.

## 3. Board State Module

A new source file `hub/src/board.rs` owns all board data access. It mirrors
`hub/src/state.rs` in every respect: directory creation, lock acquisition via
`mkdir`, atomic rename on write, temp-file naming with pid, and 5-second lock-steal
timeout.

The Rust `BoardState` struct matches the `state.json` schema written by `board.mjs`
exactly:
- `projects` â€” map from ULID string to Project (`id`, `name`, `createdAt`)
- `columns` â€” map from ULID string to Column (`id`, `projectId`, `name`, `sort` as
  integer)
- `cards` â€” map from ULID string to Card (`id`, `columnId`, `title`, `body`, `sort`
  as integer, `createdAt`)
- `comments` â€” map from ULID string to Comment (`id`, `cardId`, `author`, `body`,
  `createdAt`)
- `rev` â€” non-negative integer, incremented on every committed mutation

All timestamp fields are RFC3339 UTC at second precision with no millisecond fraction
(e.g. `2026-06-15T10:00:00Z`), matching `nowIso()` in `board.mjs`.

The `rev` field defaults to 0 on load when missing. The load function uses the same
`emptyState` fallback as `board.mjs`: absent or corrupt file returns a fresh empty
state.

The global cwd index from `board.mjs` is not ported. Hub is cwd-scoped; the
multi-cwd dropdown is not needed.

## 4. Board Verbs

All 11 board verbs are implemented in `hub/src/board.rs` as methods on a `Board`
struct, parity to `board.mjs`. Request and response shapes are identical so existing
`.board/state.json` files are usable without migration.

Parity requirements:

`project_resolve` â€” requires `name`. Returns existing project if name matches, or
creates one with fresh ULID and current ISO timestamp.

`project_list` â€” returns all projects sorted by `createdAt` ascending.

`board_get` â€” requires `projectId`. Returns project plus columns (filtered by
`projectId`, sorted by `sort` then `id`) each carrying cards (filtered by
`columnId`, same sort), each annotated with `commentCount`.

`column_create` â€” requires `projectId` and `name`. Sort is one past the max current
sort among sibling columns, or 0 if none.

`column_delete` â€” requires `id`. Returns `{ status: "unknown" }` if absent.
Cascades: deletes all cards in column, then all comments on those cards.

`card_create` â€” requires `columnId` and `title`. `body` defaults to empty string.
Sort is one past max among siblings.

`card_update` â€” requires `id`. Updates `title` and/or `body` if provided.

`card_move` â€” requires `id` and `toColumnId`. `newIndex` defaults to end. Sets
`columnId`, splices into destination column at clamped index, renumbers all cards in
destination column from 0.

`card_delete` â€” requires `id`. Returns `{ status: "unknown" }` if absent. Deletes
card and all its comments.

`comment_add` â€” requires `cardId`, `author`, `body`. Returns created comment.

`comment_list` â€” requires `cardId`. Returns comments sorted by `createdAt` ascending,
then `id` for tie-breaking.

## 5. WebSocket Board UI

Axum WebSocket support (`axum::extract::ws`) provides the `/ws` endpoint. On
connect: push full current state as JSON containing `projects`, `columns`, `cards`,
`comments` (from `.board/state.json`) and `roster` (snapshot of live agents from
`.mesh/state.json`).

On any board or mesh mutation: push a full state snapshot to all connected WebSocket
clients. Full snapshots (not diffs) for simplicity; payload is small for a personal
board.

Two broadcast channels: one fires on any board mutation (`Board::on_mutate`), one
fires on any mesh mutation (any verb that writes `.mesh/state.json`). Both carry
zero-payload tokens. The WS handler subscribes to both and re-reads both state files
on notification.

The embedded HTML page lives at `hub/src/board_ui.html` and is compiled into the
binary via `include_str!("board_ui.html")`. Served by `GET /` with
`text/html; charset=utf-8`.

The page has three visual regions: kanban (columns left-to-right with cards), mesh
roster panel (live agents and their claims), and header with project name and
live-status indicator. Vanilla JavaScript `WebSocket` connects to
`ws://localhost:7777/ws`. On message: re-render. On close: reconnect with exponential
backoff.

Drag-and-drop calls convenience REST routes at `POST /api/<verb>` (thin wrappers
over board methods, not MCP-accessible, not in tool list).

## 7. SessionStart Hook

New file `.claude/hooks/hub-ensure.sh` (bash script):

Logic: probe `curl -sf http://localhost:7777/health` with 2-second timeout. If
succeeds, exit 0. If fails, launch:
`nohup "${CLAUDE_PLUGIN_ROOT}/hub/target/release/hub" serve >/dev/null 2>&1 &`
Then poll health up to 10 times with 500ms pauses (5 seconds total). If healthy,
exit 0. If not healthy after 10 attempts, emit warning to stderr and exit 0 â€” the
hook must never abort a session.

The `/health` endpoint responds with `{ "hub": "machine-hub", "version": "0.8.0" }`
and status 200. The hook checks for the `machine-hub` identity string to guard
against a foreign daemon on port 7777.

`hooks.json` `SessionStart` array gains:
`{ "type": "command", "command": "bash \"${CLAUDE_PLUGIN_ROOT}/.claude/hooks/hub-ensure.sh\"" }`

The script is idempotent: concurrent session starts result in only one daemon binding
the port; the others see the daemon healthy.

- `project_list` ordering
- `board_get` structure (column count, card count, comment counts)

load through Rust, round-trip through save, compare structurally as parsed JSON.

- `POST /mcp tools/call board_get` returns empty columns array
- `GET /ws` first message contains `projects`, `columns`, `cards`, `comments`, `roster`

WebSocket push test: open WS client, call `project_resolve` via `POST /mcp`, verify
